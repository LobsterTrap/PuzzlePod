# PuzzlePod -- ISO 27001 Annex A Control Mapping

This document maps ISO/IEC 27001 Annex A controls to PuzzlePod features that implement or support each control. Controls are organized using both the **ISO 27001:2013** Annex A structure (A.5-A.18) and the **ISO 27001:2022** thematic structure (A.5-A.8) for cross-referencing. Where a 2022 control replaces or merges 2013 controls, both references are noted.

PuzzlePod primarily addresses **Technological controls**. Organizational, People, and Physical controls are the responsibility of the deploying organization; relevant supportive features are noted where applicable.

---

## ISO 27001:2013 Annex A Mapping

The following sections map to the ISO 27001:2013 Annex A clauses specifically requested for compliance assessment.

### A.5 -- Information Security Policies (2013)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.5.1.1 | Policies for information security | Agent profiles (YAML) and governance policies (OPA/Rego) provide machine-enforceable security policy. Profile validation against JSON schema ensures policy correctness. Version-controlled policy files enable change tracking. |
| A.5.1.2 | Review of the policies for information security | `puzzlectl profile list` and `puzzlectl profile show` enable review of active profiles. `puzzlectl policy test` validates policies before deployment. Audit trail provides evidence of policy enforcement effectiveness. |

### A.8 -- Asset Management (2013)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.8.1.1 | Inventory of assets | `puzzlectl branch list` enumerates all active branches (agent workspaces). `puzzlectl agent list` enumerates all active agents. Branch metadata includes base path, profile, creation time, and resource usage. |
| A.8.1.2 | Ownership of assets | Each branch is owned by a specific agent ID. Agent profiles define the owning user (POSIX UID). Audit trail attributes all actions to specific agent/user identifiers. |
| A.8.1.3 | Acceptable use of assets | Agent profiles define acceptable use boundaries: filesystem paths, executables, network domains, resource limits. OPA/Rego governance policies enforce acceptable use at commit time. |
| A.8.2.1 | Classification of information | Landlock denylist classifies sensitive paths (credentials, keys, system configuration). OPA policies classify changeset content (sensitive file patterns, credential patterns, high-entropy strings). |
| A.8.2.3 | Handling of assets | OverlayFS isolation ensures agent modifications are contained in the upper layer. Governance gate reviews all changes before they affect the base filesystem. Branch rollback removes all agent data with zero residue. |
| A.8.3.2 | Disposal of assets | Branch rollback (`rm -rf` of OverlayFS upper layer) is complete disposal. PID namespace destruction terminates all processes. Network namespace destruction closes all sockets. No residue remains after rollback. |

### A.9 -- Access Control (2013)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.9.1.1 | Access control policy | Three pre-defined profiles (restricted, standard, privileged) implement tiered access control. Custom profiles enable per-agent policy. Profiles are declarative YAML validated against JSON schema. |
| A.9.1.2 | Access to networks and network services | Network namespace isolation with four modes: Blocked (no access), Gated (allowlisted domains via proxy), Monitored (full access with logging), Unrestricted (dev/test only). Domain allowlists restrict reachable endpoints. |
| A.9.2.1 | User registration and de-registration | Agents registered via D-Bus `CreateBranch` API. Each agent assigned a unique UUID. De-registration via `puzzlectl agent kill` and `puzzlectl branch rollback`. Lifecycle events logged to audit. |
| A.9.2.2 | User access provisioning | Profile assignment at branch creation defines all access rights. Profile changes take effect on next branch creation. No runtime privilege escalation possible (Landlock and seccomp are irrevocable). |
| A.9.2.3 | Management of privileged access rights | Agents run with zero Linux capabilities. `puzzled` runs with minimal required capabilities. SELinux `neverallow` rules prevent privilege escalation. seccomp static deny blocks all privilege escalation syscalls. |
| A.9.2.6 | Removal or adjustment of access rights | Branch rollback removes all agent access instantly. PID namespace kill terminates all processes. `puzzlectl agent kill` provides manual termination. No residual access after termination. |
| A.9.4.1 | Information access restriction | Landlock LSM enforces filesystem access restrictions: `read_allowlist`, `write_allowlist`, `denylist`. Restrictions are irrevocable and kernel-enforced. Denylist entries override allowlist entries. |
| A.9.4.2 | Secure log-on procedures | Agents identified by system-generated UUID at branch creation. D-Bus authentication via system bus credentials (UID/GID verification). No password-based authentication. |
| A.9.4.4 | Use of privileged utility programs | seccomp USER_NOTIF gates all `execve` calls against profile `exec_allowlist`. BPF LSM enforces per-cgroup exec counting and rate limiting. Only explicitly allowed binaries can be executed. |

### A.12 -- Operations Security (2013)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.12.1.1 | Documented operating procedures | `puzzled.conf` documents all daemon configuration. Profile YAML files document per-agent access policy. Rego files document governance rules. All are machine-readable and machine-enforceable. |
| A.12.1.2 | Change management | OPA/Rego governance gate reviews all agent changes before commit. Policy violations block commits (fail-closed). IMA-signed manifests provide tamper-evident change records. Four-phase migration path for controlled rollout. |
| A.12.1.3 | Capacity management | cgroups v2 enforce per-agent resource limits: `memory.max`, `cpu.max`, `io.max`, `pids.max`. XFS project quotas enforce per-branch storage and inode limits. Prometheus metrics expose real-time capacity usage. `puzzle.slice` enforces collective limits. |
| A.12.1.4 | Separation of development, testing, and operational environments | OverlayFS branches provide per-agent isolated environments. Agent changes never affect the base filesystem until governance-approved commit. Rollback provides clean separation between agent state and production state. |
| A.12.2.1 | Controls against malware | Governance policies detect persistence mechanisms (cron jobs, systemd units, init scripts). seccomp exec gating prevents execution of unauthorized binaries. BPF LSM enforces exec rate limits. OPA policies scan changesets for malicious patterns. |
| A.12.4.1 | Event logging | Linux Audit subsystem logs all agent lifecycle events. fanotify logs file access events. seccomp USER_NOTIF logs exec/connect/bind decisions. Prometheus metrics log operational statistics. |
| A.12.4.2 | Protection of log information | Audit trail stored via Linux Audit subsystem (protected by auditd). IMA-signed manifests provide tamper evidence. `puzzled` audit directory protected by SELinux type (`puzzlepod_branch_t`). Agents cannot access or modify audit logs. |
| A.12.4.3 | Administrator and operator logs | All `puzzled` administrative actions logged: branch create, commit, rollback, policy reload, agent kill. `puzzlectl audit export` produces structured admin event data. D-Bus method calls logged with caller credentials. |
| A.12.4.4 | Clock synchronization | Audit events use system clock. IMA manifests include signed timestamps. WAL entries include monotonic and wall-clock timestamps. Relies on system NTP (out of scope for PuzzlePod). |
| A.12.5.1 | Installation of software on operational systems | seccomp USER_NOTIF gates all `execve` calls. Only binaries in the profile `exec_allowlist` can execute. OPA governance blocks installation of persistent software (cron, systemd units). OverlayFS isolation prevents modification of base system binaries. |
| A.12.6.1 | Management of technical vulnerabilities | Eight independent defense layers ensure single-layer vulnerability does not compromise containment. Userspace-only architecture deploys on stock kernels (no custom modules). Security test suite (`tests/security/`) covers escape vectors. |

### A.14 -- System Acquisition, Development, and Maintenance (2013)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.14.1.1 | Information security requirements analysis and specification | Agent profiles define security requirements per workload type. Threat model (T1-T7) analyzes requirements for agentic workloads. Defense-in-depth architecture addresses each requirement with kernel-enforced mechanisms. |
| A.14.1.2 | Securing application services on public networks | Network gating restricts agent access to public networks. HTTP proxy mediates all outbound requests. Domain allowlists control reachable endpoints. TLS enforcement configurable per-profile. POST/PUT/DELETE queued for governance review. |
| A.14.2.1 | Secure development policy | Rust implementation provides memory safety (no buffer overflows, use-after-free, double-free). `puzzled` hardened with seccomp, SELinux, minimal capabilities. All profiles and policies validated before deployment. |
| A.14.2.2 | System change control procedures | Governance gate (OPA/Rego) reviews all agent changes before commit to base filesystem. Policy violations block changes. WAL ensures atomicity. IMA-signed manifests provide change audit trail. |
| A.14.2.3 | Technical review of applications after operating platform changes | Integration tests (`tests/integration/`) validate full fork-explore-commit lifecycle. Security tests (`tests/security/`) verify escape vector coverage. Performance benchmarks (`tests/performance/`) measure enforcement overhead. |
| A.14.2.5 | Secure system engineering principles | Defense-in-depth with eight independent kernel-enforced layers. Fail-closed default behavior. Deterministic governance (no ML or heuristics). Least privilege via Landlock, seccomp, zero capabilities. |
| A.14.2.8 | System security testing | Security test suite covers: escape testing, privilege escalation, policy bypass, namespace escape. Tests run with `sudo ./tests/security/run_all.sh`. Integration tests cover crash recovery scenarios. |

## Technological Controls (A.8)

| Control | Title | PuzzlePod Implementation |
|---|---|---|
| A.8.1 | User endpoint devices | Out of scope (endpoint management). PuzzlePod operates at the server/edge OS level. |
| A.8.2 | Privileged access rights | Agents run with zero Linux capabilities. `puzzled` runs with minimal required capabilities. SELinux confines `puzzled` to `puzzlepod_t` domain and agents to `puzzlepod_t` domain. Neverallow rules prevent privilege escalation. seccomp static deny blocks all privilege escalation syscalls. |
| A.8.3 | Information access restriction | Landlock LSM enforces filesystem access restrictions: `read_allowlist`, `write_allowlist`, `denylist`. Restrictions are irrevocable and kernel-enforced. Agent cannot modify its own Landlock ruleset. Denylist entries take precedence over allowlist entries. |
| A.8.4 | Access to source code | Landlock restricts agent file access to paths specified in the profile. Source code repositories can be excluded via `denylist` or limited via `read_allowlist`. Write access to source code goes to OverlayFS upper layer; governance gate reviews before commit. |
| A.8.5 | Secure authentication | Agents identified by system-generated UUID. D-Bus authentication via system bus credentials. POSIX UID for multi-tenancy. No password-based authentication (agents are processes, not users). |
| A.8.6 | Capacity management | cgroups v2 enforce per-agent resource limits: `memory_bytes`, `cpu_shares`, `io_weight`, `max_pids`. XFS project quotas enforce per-branch storage limits: `storage_quota_mb`, `inode_quota`. Prometheus metrics expose real-time capacity usage. |
| A.8.7 | Protection against malware | Governance policies detect persistence mechanisms in changesets (cron jobs, systemd units, init scripts). seccomp exec gating prevents execution of unauthorized binaries. BPF LSM enforces exec counting and rate limiting. Sensitive file patterns detected in committed changesets. |
| A.8.8 | Management of technical vulnerabilities | Security test suite covers escape testing, privilege escalation, and policy bypass. Eight independent defense layers ensure that vulnerability in one layer does not compromise others. Userspace-only architecture deploys on stock kernels (no custom kernel modules to patch). |
| A.8.9 | Configuration management | `puzzled.conf` is the single configuration source. Agent profiles validated against JSON schema. Policy changes tested via `puzzlectl policy test` before reload. Default configurations follow least-privilege principles. |
| A.8.10 | Information deletion | Branch rollback (`rm -rf` of OverlayFS upper layer) deletes all agent-generated data with zero residue. PID namespace destruction terminates all processes. Network namespace destruction closes all sockets. Rollback is the default action on governance failure. |
| A.8.11 | Data masking | Landlock denylist prevents agent access to sensitive data paths (`/etc/shadow`, `.ssh/`, `.env`). OverlayFS isolation ensures agents only see their own branch data. PID namespace prevents agents from reading other agents' `/proc` entries. |
| A.8.12 | Data leakage prevention | Network gating prevents unauthorized data transmission. Domain allowlists restrict outbound endpoints. HTTP proxy queues write requests (POST/PUT/DELETE) for governance review. DNS restricted to controlled resolvers. Governance policies scan changesets for credential patterns and high-entropy strings. |
| A.8.13 | Information backup | WAL-based commit ensures crash-safe persistence. Base filesystem is never modified during agent execution (OverlayFS isolation). Rollback restores the base filesystem to its pre-agent state. Out-of-scope: backup of the base filesystem itself is the organization's responsibility. |
| A.8.14 | Redundancy of information processing facilities | `puzzled` auto-restarts via systemd. WAL recovery ensures in-flight commits are resolved (committed or rolled back) on restart. Orphaned branch cleanup on startup. Kernel-enforced restrictions persist independently of daemon availability. |
| A.8.15 | Logging | Linux Audit subsystem logs all agent lifecycle events. fanotify logs file access events. seccomp USER_NOTIF logs exec/connect/bind decisions. Prometheus metrics log operational statistics. `puzzlectl audit export` provides structured log export. |
| A.8.16 | Monitoring activities | fanotify behavioral monitoring detects anomalous file access patterns in real time. BPF LSM monitors exec call frequency. cgroup monitoring detects resource pressure and OOM events. Prometheus metrics enable dashboard and alerting integration. |
| A.8.17 | Clock synchronization | Audit events use system clock. IMA manifests include signed timestamps. Relies on system NTP configuration (out of scope for PuzzlePod). |
| A.8.18 | Use of privileged utility programs | seccomp USER_NOTIF gates all `execve` calls against profile-defined `exec_allowlist`. BPF LSM enforces per-cgroup exec counting. Governance policy blocks installation of persistent utilities (cron, systemd units). |
| A.8.20 | Networks security | Network namespace provides per-agent network isolation. nftables rules enforce per-agent network policy. Four modes: Blocked, Gated, Monitored, Unrestricted. Domain allowlists restrict outbound connections. |
| A.8.21 | Security of network services | HTTP proxy (`puzzle-proxy`) mediates agent access to external services. GET/HEAD pass-through (read-only). POST/PUT/DELETE queued for governance review. TLS enforcement configurable. |
| A.8.22 | Segregation of networks | Each agent operates in its own network namespace with isolated network stack. No cross-agent network communication possible. nftables blocks all traffic not explicitly allowed. |
| A.8.23 | Web filtering | HTTP proxy enforces domain allowlists for agent web access. Domains not in the allowlist are connection-refused. DNS queries routed to controlled resolver only. |
| A.8.24 | Use of cryptography | IMA manifest signing uses kernel keyring cryptographic keys. SHA-256 checksums on all changeset files. WAL integrity protected by checksums. |
| A.8.25 | Secure development lifecycle | Rust implementation provides memory safety guarantees. Security test suite tests escape vectors. Integration tests cover fork-explore-commit lifecycle. Profile and policy validation prevents configuration errors. |
| A.8.26 | Application security requirements | Agent profiles define security requirements per application. OPA/Rego policies define commit-time security requirements. Both are declarative, testable, and version-controlled. |
| A.8.28 | Secure coding | Rust language eliminates memory safety vulnerabilities (buffer overflows, use-after-free, double-free). `puzzled` hardened with seccomp (57 blocked escape-vector syscalls + 4 USER_NOTIF-gated; 58 on x86_64), SELinux confinement, minimal capabilities. |
| A.8.31 | Separation of development, test and production environments | OverlayFS branches provide per-agent isolated environments. Agent changes never affect the base filesystem until governance-approved commit. Rollback provides clean separation between test and production state. |
| A.8.32 | Change management | Governance gate (OPA/Rego) reviews all agent changes before commit. Policy violations block changes. IMA-signed manifests provide change audit trail. Four-phase migration path provides controlled change rollout. |
| A.8.33 | Test information | Agent testing occurs in isolated OverlayFS branches. Test changes do not affect production (base) filesystem. Rollback removes test artifacts completely. |
| A.8.34 | Protection of information systems during audit testing | Agents operate in isolated branches during any audit or test scenario. Base filesystem protected by OverlayFS CoW semantics. cgroup limits prevent resource impact on other workloads. |

## Summary

PuzzlePod provides strong technical controls for ISO 27001:2022 Annex A, particularly in the **Technological Controls (A.8)** category:

| Area | Key Controls Addressed | Primary Mechanisms |
|---|---|---|
| Access Control | A.8.2, A.8.3, A.8.4, A.8.5 | Landlock, seccomp, SELinux, zero capabilities |
| Data Protection | A.8.10, A.8.11, A.8.12, A.8.13 | OverlayFS isolation, network gating, governance scanning, rollback |
| Monitoring and Logging | A.8.15, A.8.16 | Linux Audit, fanotify, BPF LSM, Prometheus |
| Network Security | A.8.20, A.8.21, A.8.22, A.8.23 | Network namespaces, nftables, HTTP proxy, domain allowlists |
| Malware and Vulnerability | A.8.7, A.8.8 | Governance policies, exec gating, defense-in-depth |
| Change Management | A.8.9, A.8.32 | Profile validation, governance gate, IMA signing |
| Secure Development | A.8.25, A.8.26, A.8.28 | Rust memory safety, security tests, profile/policy validation |

Organizational controls (A.5), People controls (A.6), and Physical controls (A.7) remain the responsibility of the deploying organization. PuzzlePod provides the enforcement infrastructure and audit evidence that support these organizational controls.
