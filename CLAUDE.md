# CLAUDE.md — Technical Context for AI Assistants

## Project Overview

**PuzzlePod** is a userspace governance daemon (`puzzled`) and CLI (`puzzlectl`) that add **automated governance** to AI agent workloads running in containers on Linux. It composes with Podman and systemd — not reinvents them. Container isolation (namespaces, cgroups, seccomp, SELinux) is handled by Podman/crun as usual; `puzzled` adds the governance layer on top: OPA/Rego policy evaluation on changesets, automated commit/rollback decisions, and optional runtime mediation of specific syscalls. The core concept is **Fork, Explore, Commit** — agents execute in isolated copy-on-write filesystem branches (OverlayFS), their changes are reviewed by the governance policy engine, and only approved changes are committed. Failures and rejections result in clean rollback with zero residue. No kernel modifications required.

This project targets RHEL 10+, Fedora 42+, and CentOS Stream 10 on x86_64 and aarch64. It is designed to operate on both data center servers and resource-constrained edge devices (4GB+ RAM). Safety certification compatibility (IEC 61508, ISO 26262) is a design requirement.

## Enforcement Model: Userspace Configures, Kernel Enforces

`puzzled` is a userspace governance daemon that **configures** kernel enforcement mechanisms on behalf of agent processes. Once configured, the enforcement is performed by the kernel and is **irrevocable by the agent**.

There are two operating modes for container lifecycle. Direct mode is implemented; Podman-native mode is proposed (see `docs/podman_puzzled_architecture.md`):

| Mechanism | Configured by (direct mode) | Configured by (Podman-native mode, proposed) | Enforced by (kernel) | Survives puzzled crash? |
|---|---|---|---|---|
| Landlock ruleset | puzzled applies via `landlock_restrict_self()` | `puzzle-init` shim inside container (rules generated by puzzled) | Kernel LSM hook on every file access | **Yes** — attached to process, not daemon |
| seccomp-BPF filter | puzzled loads via `seccomp()` | crun loads from OCI seccomp profile (generated by puzzled) | Kernel checks on every syscall | **Yes** — irrevocable once loaded |
| PID namespace | puzzled creates via `clone3()` | Podman/crun | Kernel process table isolation | **Yes** — namespace persists with process |
| cgroup limits | puzzled writes to cgroup fs | Podman (via `--memory`, `--cpus`, etc.) | Kernel scheduler + OOM killer | **Yes** — cgroup persists independently |
| Mount namespace | puzzled creates via `clone3()` | Podman/crun | Kernel VFS layer | **Yes** — namespace persists with process |

This is categorically different from application-level guardrails (prompt engineering, SDK permission lists) where the agent process could bypass the check by calling the underlying syscall directly. **Containment is kernel-enforced and agent-irrevocable; governance (OPA/Rego policy evaluation, commit approval) is userspace logic within `puzzled`.**

See PRD Section 2.3 and Section 5.4.4 for the full rationale.

## What Is an Agentic Workload (in Linux Primitives)

An **agentic workload** is a process tree where the syscall sequence — which files are opened, which binaries are executed, which network connections are initiated — is determined at runtime by an external inference engine (LLM) rather than by the compiled program logic alone.

From the kernel's perspective, an agentic process is indistinguishable from any other process — there is no "agent bit" in `task_struct`. The kernel sees a process, a UID, capabilities, and a syscall number. An `execve("rm", ["-rf", "/"])` from a hallucinating LLM looks identical to the same syscall from a human typing in a shell.

**What makes agentic workloads unique:**

| Property | Traditional Process | Agentic Workload |
|---|---|---|
| Syscall sequence | Determined by compiled code | Determined by LLM inference at runtime |
| Files accessed | Known at design time | Chosen by LLM at runtime |
| Subprocesses spawned | Fixed set (e.g., worker pool) | Arbitrary binaries (`gcc`, `curl`, `rm`, …) |
| Network endpoints | Configured statically | Chosen by LLM at runtime |
| Trust model | Trusted code, trusted control flow | Trusted identity, **untrusted control flow** |

An agentic workload combines the **broad syscall surface of an interactive shell**, the **autonomous execution of a daemon**, and the **untrusted control flow of a container**. No existing Linux isolation primitive addresses this combination — which is the gap `puzzled` and the Branch Context fill.

See PRD Section 2.5 for the full definition with syscall-level detail and workload comparison table.

## Architectural Decision: Userspace-First (v2.0)

**No kernel modifications.** After detailed analysis (see `Kernel_vs_userspace.md`), the project determined that existing upstream kernel primitives provide ~85-90% of required agent containment functionality. The remaining ~10-15% (copy-up pollution in diff, non-atomic commit, O(n) upper-layer walk) is addressed in userspace with acceptable tradeoffs.

**Key existing primitives used (all upstream, unmodified):**

| Primitive | Kernel Version | Function |
|---|---|---|
| Landlock LSM | 5.13+ (ABI v4: 6.7+) | Unprivileged, irrevocable filesystem + network ACL |
| BPF LSM | 5.7+ | Programmable per-cgroup security hooks (exec counting, rate limiting) |
| seccomp-BPF | 3.5+ (USER_NOTIF: 5.0+) | Static deny for escape vectors + dynamic daemon-mediated gating for execve/connect/bind |
| PID namespaces | 3.8+ | Process isolation + reliable termination |
| Mount namespaces | 2.4.19+ | Per-agent filesystem view |
| Network namespaces | 2.6.29+ | Per-agent network isolation |
| cgroups v2 | 4.5+ | Resource limits (memory, CPU, I/O, PIDs) |
| cgroup.freeze | 5.2+ | Freeze agent for TOCTOU-free diff reading |
| OverlayFS | 3.18+ | Copy-on-write filesystem branching |
| XFS project quotas | 3.0+ | Per-branch storage/inode limits on upper layer |
| pidfd | 5.3+ | Race-free process lifecycle management |
| clone3() | 5.3+ | Modern process creation with namespace flags |
| fanotify | 2.6.37+ (FID: 5.1+) | Real-time file access monitoring + behavioral triggers |
| Linux Audit | 2.6+ | Security event logging |
| IMA | 2.6.30+ | Integrity measurement for changeset signing |
| SELinux | 2.6+ | Mandatory label-based access control |

**What we build (all userspace):**
- `puzzled` — governance daemon that composes these primitives into agent sandboxes
- `puzzlectl` — CLI for branch management and governance
- `puzzle-podman` — bash wrapper for Podman-native mode (delegates to `podman` + `puzzled`)
- `puzzle-hook` — **(proposed)** OCI runtime hook for Podman-native mode (Rust binary, ~500 lines)
- `puzzle-init` — **(proposed)** Landlock shim entrypoint for Podman-native mode (static binary, ~200 lines)
- BPF LSM programs — small eBPF programs for exec counting/rate limiting
- OPA/Rego policies — governance rules evaluated at commit time
- SELinux policy module — type enforcement for agent domains

**What we do NOT build:**
- No kernel modules (original AgentFS, AgentGuard LSM, agent cgroup controller designs preserved in Appendix D of PRD)
- No new syscalls or ioctls
- No kernel patches (optional narrow OverlayFS change log deferred to Phase 3, conditional on evidence)

**Why not kernel extensions?** See `Kernel_vs_userspace.md` for full analysis. Summary: upstream acceptance probability for new LSMs is "very low", new cgroup controllers is "low", new syscalls is "low". Only an OverlayFS change log has "medium-high" acceptance probability, and it's optional.

## Architecture

### Two Operating Modes

PuzzlePod supports two operating modes for container lifecycle management. Both provide the same governance guarantees (Fork, Explore, Commit with OPA-gated approval); they differ in who creates the container:

| Mode | Container created by | Use case | Root required? | Mac/Windows? | Status |
|---|---|---|---|---|---|
| **Direct mode** | puzzled via `clone3()` | Edge devices, safety-critical, minimal TCB | Yes | No | **Implemented** |
| **Podman-native mode** | Podman/crun (unmodified) | Data center, developer workstations, CI/CD | No (rootless supported) | Yes (via `podman machine`) | **Proposed** (see PRD) |

In **Podman-native mode** (proposed), puzzled becomes a governance-only daemon. Podman handles container lifecycle (namespaces, cgroups, networking, OCI images), while puzzled provides the unique governance layer (branch semantics, OPA policy, seccomp USER_NOTIF mediation, Landlock, BPF LSM, fanotify, audit). Zero Podman source code changes are required — all integration uses documented extension points (OCI runtime hooks, seccomp profiles, container annotations, bind mounts).

See `docs/podman_puzzled_architecture.md` for the full Podman-native architecture specification.

### Execution Flow (Fork, Explore, Commit)

#### Direct Mode

1. **Fork:** `puzzled` calls `clone3(CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWIPC | CLONE_NEWUTS | CLONE_NEWCGROUP | CLONE_PIDFD)` → child PID 1 in new PID namespace. Network namespace is joined via `setns()` into a pre-created named netns (not `CLONE_NEWNET`). `puzzled` gets `pidfd` for race-free monitoring. Sets up OverlayFS mount, Landlock ruleset, seccomp-BPF filter (static deny + USER_NOTIF for execve/connect), cgroup scope, BPF LSM hooks, fanotify marks.

2. **Explore:** Agent runs in sandbox. OverlayFS captures all writes in upper layer. Landlock restricts filesystem access. seccomp USER_NOTIF gates execve/connect through puzzled. seccomp static filter blocks escape syscalls. cgroup enforces resource limits. XFS project quotas enforce storage/inode limits on upper layer. fanotify monitors file access patterns for behavioral triggers.

3. **Commit/Rollback:**
   - `puzzled` calls `cgroup.freeze` to freeze agent processes (TOCTOU protection)
   - Walks OverlayFS upper layer to generate diff (checksum filtering for copy-up artifacts)
   - Evaluates OPA/Rego governance policy against diff
   - If approved: WAL-based crash-safe commit (log → execute → mark complete)
   - If rejected: discard upper layer directory (zero residue)
   - `pidfd` monitoring detects agent exit; PID namespace kill cleans up all processes

#### Podman-Native Mode (Proposed)

1. **Fork:** `puzzle-podman` wrapper calls puzzled (D-Bus) to create branch (OverlayFS upper layer), generate seccomp profile (with `SCMP_ACT_NOTIFY` + `listenerPath`), and generate Landlock rules file. Then runs `podman run` with the branch merged dir as a bind mount, the seccomp profile, the `puzzle-init` shim as entrypoint, and `run.oci.handler=puzzlepod` annotation. crun creates the container (namespaces, cgroups, rootfs), loads the seccomp filter, and sends the notification fd to puzzled via the `listenerPath` socket. The `puzzle-init` shim applies Landlock (`landlock_restrict_self()`) then execs the real command. The OCI `createRuntime` hook triggers `puzzle-hook`, which calls puzzled to attach BPF LSM and start fanotify monitoring.

2. **Explore:** Identical to direct mode — same kernel enforcement primitives, same governance.

3. **Commit/Rollback:** On container exit, the OCI `poststop` hook triggers `puzzle-hook`, which calls puzzled to run governance (freeze → diff → OPA evaluate → commit/rollback). Same WAL-based crash-safe commit.

### Seccomp Split Strategy

Syscalls are split into two tiers to avoid unnecessary overhead:

| Tier | Mechanism | Latency | Used For |
|---|---|---|---|
| **High-frequency, static policy** | Landlock (in-kernel) | < 1 μs | `open`, `read`, `write`, `stat` |
| **Low-frequency, dynamic policy** | seccomp `SECCOMP_RET_USER_NOTIF` | ~50-100 μs | `execve`, `connect`, `bind` |
| **Monitoring only** | fanotify (async) | Zero agent impact | File access patterns, behavioral triggers |

### Security: Defense-in-Depth (Independent Enforcement Layers)

Every escape vector is blocked by at least two independent mechanisms:

| Layer | Mechanism | Kernel Version | Configured by (direct) | Configured by (Podman-native, proposed) | Survives puzzled crash? |
|---|---|---|---|---|---|
| 0 | Landlock | 5.13+ | puzzled | `puzzle-init` shim (rules from puzzled) | **Yes** |
| 1 | seccomp-BPF | 3.5+ / 5.0+ | puzzled | crun (profile from puzzled) | **Yes** (static deny); USER_NOTIF gated calls return ENOSYS if puzzled crashes |
| 2 | PID namespace | 3.8+ | puzzled (`clone3`) | Podman/crun | **Yes** |
| 3 | Mount namespace | 2.4.19+ | puzzled (`clone3`) | Podman/crun | **Yes** |
| 4 | Network namespace | 2.6.29+ | puzzled (`setns`) | Podman/netavark | **Yes** |
| 5 | cgroups v2 | 4.5+ | puzzled | Podman | **Yes** |
| 6 | SELinux | 2.6+ | puzzled | Podman (`--security-opt label=type:puzzlepod_t`) | **Yes** |
| 7 | BPF LSM | 5.7+ | puzzled | puzzled (via OCI hook) | **Yes** |
| 8 | User namespace | 3.8+ | N/A (direct mode runs as root) | Podman (rootless mode) | **Yes** |

## Userspace Components

### puzzled — Governance Daemon

**Location:** `crates/puzzled/`

Central governance daemon managing OverlayFS branching, policy evaluation, and audit. Runs as root (system instance) or as a regular user (user instance for rootless mode).

**Key responsibilities:**
- D-Bus API on system bus (`org.lobstertrap.PuzzlePod1.Manager`) or session bus (rootless)
- **Direct mode:** Agent sandbox creation via `clone3()` + `pidfd` + namespace/cgroup/Landlock/seccomp setup
- **Podman-native mode (proposed):** Governance-only — generates seccomp profiles and Landlock rules for Podman/crun; receives seccomp notification fds via Unix socket from crun; attaches BPF LSM and fanotify via OCI hook
- OverlayFS branch management (create upper layer, mount, diff, commit, rollback)
- OPA/Rego policy engine (via `regorus`, a pure-Rust Rego evaluator) for commit governance
- WAL-based crash-safe commit (log → execute → mark complete)
- `cgroup.freeze` for TOCTOU-free diff reading
- XFS project quota setup on OverlayFS upper layer (root mode only)
- BPF LSM program loading for exec counting/rate limiting (requires `CAP_BPF`; disabled rootless)
- seccomp user notification handling for execve/connect/bind gating
- fanotify behavioral monitoring with configurable triggers (mass deletion, credential access, excessive reads); partial in rootless mode (`FAN_REPORT_FID` requires `CAP_SYS_ADMIN`)
- Agent health monitoring via `pidfd` + watchdog
- IMA-integrated changeset signing
- Linux Audit event generation
- Network side-effect gating via HTTP proxy (Phase 2)

**Process architecture:**
```
puzzled (runs as root or user)
├── Main thread: D-Bus listener + agent registration
├── Seccomp listener: Unix socket for crun notif fd (Podman-native mode, proposed)
├── Seccomp thread: Per-branch seccomp notification fd polling (epoll)
├── Fanotify thread: Per-branch fanotify fd polling (epoll); behavioral triggers
├── Policy thread: OPA evaluation engine (pooled)
├── Watchdog thread: Heartbeat monitoring + timeout enforcement
├── Cleanup thread: Branch garbage collection + journal recovery
└── Audit thread: Event serialization + signing
```

**Configuration:**
- System instance: `/etc/puzzled/puzzled.conf`
- User instance (rootless): `~/.config/puzzled/puzzled.conf` (or `$XDG_CONFIG_HOME/puzzled/puzzled.conf`)
- CLI override: `puzzled --config <path>`
- Auto-detection: `DaemonConfig::load_or_default()` checks `/etc/puzzled/puzzled.conf` first, then the user config path for non-root users, then falls back to `DaemonConfig::default_for_user()` which sets session bus, ext4, disables BPF LSM/fanotify, and uses XDG-compliant paths under `$HOME`.


**Hardening:** Minimal capabilities, SELinux-confined (`puzzlepod_t`), seccomp-BPF (configurable per-profile: Permissive mode (default) with default-allow and ~57 KillProcess escape vectors, or Strict mode with default-deny and ~152 allowed syscalls; both with 4+ USER_NOTIF-gated syscalls), no external network access.

**Fail-closed behavior:** If puzzled crashes during governance evaluation, pending commits are rolled back on restart. Landlock restrictions on agent processes survive puzzled crash (kernel-enforced, independent). In Podman-native mode (proposed), if puzzled is down when a governed container starts, the OCI hook fails and crun aborts the container start — the container never runs ungoverned. systemd restarts puzzled; it re-discovers active branches from `/var/lib/puzzled/branches/` (or `~/.local/share/puzzled/branches/` for user instance).

**Rootless degradation:** When running as a user instance without root, BPF LSM is disabled (requires `CAP_BPF`), fanotify is partial (path-based only), XFS project quotas are unavailable (use `podman --storage-opt size=` instead), and kernel OverlayFS is replaced by fuse-overlayfs (~15-20% I/O overhead vs ~5-10%). Landlock, seccomp (static deny + USER_NOTIF), OPA policy, and the audit chain work fully in rootless mode.

### puzzle-sim-worker — Sandbox Executor

**Location:** `crates/puzzlectl/src/bin/puzzle_sim_worker.rs`

Helper binary spawned inside sandboxed branches by `activate_branch()` during `puzzlectl sim --sandbox` runs. It executes scenario actions (file writes, execs, network attempts) against the overlay-merged root from within the full sandbox (namespaces, seccomp, Landlock, cgroup). The exit code signals pass/fail to the outer `SimEngine`.

**Not a user-facing tool.** It is invoked automatically by the simulator engine. The Containerfile copies it into the image at `/usr/local/bin/puzzle-sim-worker`, making it part of the deployment surface. Gated behind the `sim` Cargo feature — compiles to a no-op stub without it.

### Podman Integration (Podman-Native Mode) — Partially Implemented

**Location:** `podman/` (wrapper). Proposed new components: `crates/puzzle-hook/` (OCI hook), `puzzle-init/` (Landlock shim) — see `docs/podman_puzzled_architecture.md`.

**Zero Podman source code changes.** All integration via documented, stable extension points.

**Implemented:**
- **`puzzle-podman`** (`podman/puzzle-podman`, 294 lines bash) — wrapper that delegates to `podman run` + `puzzlectl` D-Bus. Handles branch creation, bind-mount of merged dir, and interactive commit/rollback on exit.
- **OCI hook** (`podman/hooks/puzzle-branch-hook.sh`, 75 lines bash) — basic `createRuntime` hook triggered by `org.lobstertrap.puzzlepod.branch` annotation.

**Proposed (not yet implemented):**
- **`puzzle-hook`** (`crates/puzzle-hook/`, ~500 lines Rust) — compiled OCI runtime hook replacing the bash hook, triggered by `run.oci.handler=puzzlepod` annotation at `createRuntime` (attach BPF LSM, start fanotify, register container PID) and `poststop` (trigger governance evaluation) stages.
- **`puzzle-init`** (~200 lines static binary) — Landlock shim entrypoint bind-mounted into the container. Reads Landlock rules from `/run/puzzlepod/landlock.json`, calls `landlock_restrict_self()` (irrevocable), then `execve`s the real command. Zero ongoing overhead.
- **Enhanced wrapper** (~350-400 lines bash) — seccomp profile generation, Landlock rules generation, annotation injection, podman machine detection (Mac/Windows).
- **Quadlet** — standard `.container` unit files with `Annotation=run.oci.handler=puzzlepod`, `SecurityOpt=seccomp=...`, and branch bind mounts. No custom Quadlet directives needed.

## Governance Policies

**Location:** `policies/`

### Agent Profiles (`policies/profiles/`)

YAML files defining per-agent access control:
- `restricted.yaml` — Minimal access, no network, small quotas
- `standard.yaml` — Project-scoped access, gated network, standard quotas
- `privileged.yaml` — Broad access, monitored network, large quotas

Profiles define: filesystem read/write allowlists and denylists, executable allowlists, process limits, network mode and domain lists, Linux capabilities, behavioral trigger configuration, fail mode (for safety-critical deployments).

### Commit Rules (`policies/rules/`)

OPA/Rego policies evaluated at commit time:
- No sensitive files (credentials, SSH keys, `.env`) in changeset
- No persistence mechanisms (cron jobs, systemd units)
- No executable permission changes
- Total change size within limits
- No system file modifications

## Functional Safety

For deployment in safety-critical physical systems (vehicles, robots, drones, industrial controllers), the containment framework provides:

- **Actuator gating** — extends the network gating pattern: read sensors always allowed, write actuator commands validated against a safety envelope before execution
- **Real-time profile** — uses only in-kernel primitives with bounded WCET (Landlock, static seccomp, nftables, cgroups); no daemon-mediated decisions on the critical path
- **Configurable fail modes** — fail-closed (default), fail-silent (hold last safe state), fail-operational (reduced capability fallback), fail-safe-state (controlled stop/return to base)
- **Determinism guarantee** — all in-kernel enforcement is deterministic with < 1 μs WCET; userspace governance (seccomp USER_NOTIF, OPA evaluation) has unbounded worst-case latency and is disabled in the real-time profile

**Critical scope boundary:** The containment framework supplements but does NOT replace a certified safety controller. A deterministic, independently certified safety controller must sit between the AI agent and physical actuators.

**TCB and mode selection:** Safety-critical deployments should use direct mode (not Podman-native) to minimize the trusted computing base. Podman-native mode adds Podman, crun, and conmon to the TCB, increasing certification evidence burden for IEC 61508 / ISO 26262.

See PRD Section 18 for the full functional safety analysis, hazard categories, and certification mapping.

## Performance Targets

| Operation | x86_64 Target | aarch64 Target |
|---|---|---|
| Branch creation (OverlayFS upper layer) | < 50ms | < 100ms |
| Container start (Podman-native, additional) | ~100-200ms | ~100-200ms |
| File I/O overhead (kernel OverlayFS) | < 10% | < 10% |
| File I/O overhead (fuse-overlayfs, rootless) | ~15-20% | ~15-20% |
| Branch commit (1K files, WAL) | < 2s | < 3s |
| Branch rollback | < 10ms | < 10ms |
| Landlock check | < 1 μs | < 1 μs |
| BPF LSM check | < 1 μs | < 1 μs |
| seccomp USER_NOTIF (per call) | ~50-100 μs | ~50-100 μs |
| Concurrent branches | 64 | 8 (edge) |
| puzzled memory | < 50MB + 5MB/branch | < 30MB + 3MB/branch |

**Known structural penalties of userspace composition** (see PRD Section 17.5 for full analysis):
- seccomp USER_NOTIF: 50-100x slower than in-kernel LSM hook per intercepted call (mitigated by only intercepting low-frequency syscalls)
- Upper-dir walk at commit: O(n) vs. O(1) for hypothetical in-kernel tracker (mitigated by fanotify incremental tracking in Phase 2)
- Commit write amplification: 2x I/O vs. in-kernel rename (mitigated by `renameat2(RENAME_EXCHANGE)`)
- For typical agent workloads, aggregate overhead is tens of milliseconds — within the noise of LLM inference latency

## Key Design Principles

1. **Zero kernel modifications** — compose existing upstream primitives; no new syscalls, LSMs, or cgroup controllers
2. **The kernel enforces, userspace decides** — puzzled configures kernel enforcement (Landlock, seccomp, namespaces, cgroups); the kernel enforces irrevocably; puzzled evaluates policy (OPA/Rego) in userspace
3. **Defense in depth** — Landlock, BPF LSM, seccomp, SELinux, namespaces, cgroups are independent layers; each survives failure of the others
4. **Fail closed** — if governance cannot be determined, rollback (not commit); configurable fail modes for safety-critical deployments
5. **Deterministic behavior** — no ML, heuristics, or probabilistic decisions in governance path
6. **Composable with existing stack** — SELinux, audit, Podman, systemd continue unchanged

## Operational Considerations

See PRD Section 24 and `docs/podman_puzzled_architecture.md` for full details on:

- **Packaging:** Userspace-only RPM packages (`puzzled`, `puzzlectl`, `puzzled-selinux`, `puzzled-profiles`, `puzzled-policies`; proposed: `puzzle-hook`, `puzzle-init`). Meta-packages: `puzzlepod`, `puzzlepod-minimal`, `puzzlepod-edge`. Podman-native mode (proposed) adds runtime dependency: `podman >= 5.0`, `crun >= 1.14`. No kernel modules, no DKMS, no Secure Boot signing complexity.
- **Rootless operation:** puzzled user instance on D-Bus session bus; branches stored in `~/.local/share/puzzled/branches/`. Works in both direct mode (via `scripts/dev-setup-user.sh` and fuse-overlayfs) and Podman-native mode. Auto-detected when running as non-root (UID != 0). BPF LSM and full fanotify disabled; Landlock, seccomp, OPA policy fully operational. See `demo/run_demo_rootless.sh` for a complete walkthrough.
- **Mac/Windows support:** Via `podman machine` — `puzzle-podman` wrapper detects podman machine and proxies puzzled calls through the VM via SSH. Full kernel primitive support inside the VM.
- **Migration path:** Four adoption phases: Monitor Only → Audit Mode → Enforce Mode → Full Governance.
- **Agent framework integration:** Three options — transparent (`puzzle-podman run`), native (Rust `puzzled-client` crate), or CLI (`puzzlectl branch exec` for direct mode).
- **IDE integration:** VS Code Dev Containers (set `dockerPath` to `puzzle-podman`), JetBrains Gateway, GitHub Codespaces (puzzled as sidecar).
- **Observability:** Prometheus metrics, OpenTelemetry tracing.
- **Multi-tenancy:** POSIX UID-based identity, SELinux MCS labels, per-user cgroup slices.
- **Upgrades:** Zero-downtime restart via state serialization, WAL-based crash recovery.
